# v2 알려진 한계 (Known Limitations)

이 문서는 v1에서 의도적으로 단순화했거나 구조적 제약으로 완전 지원을 미룬 항목을
정리합니다. 각 항목은 "현재 동작 → 한계 → v2 방향"으로 기술합니다. 손실은 모두
`LossReport`로 기록되므로 다운스트림에서 감지·분기할 수 있습니다.

## 이번 변경에서 보강된 항목

다음은 "조용한 손실"이었으나 무손실 보강(텍스트 구제) 또는 감사 기록 추가로
개선된 항목입니다.

- **하이퍼링크 표시 텍스트 (HWP3)** — `Control::Hyperlink`는 표시 텍스트를 버리지
  않고 평문 블록으로 구제합니다. URL/링크 의미는 여전히 `LossEntry`로 기록됩니다.
- **덧말(Ruby) / 글자겹침(CharOverlap)** — 텍스트/글자를 평문 블록으로 구제합니다.
- **개요 수준 7+ 클램프** — 6으로 평탄화될 때 `LossEntry`를 남깁니다.
- **폰트 손실 보고** — 7개 언어 슬롯(한글/영문/한자/일어/기타/기호/사용자)을 모두
  검사합니다(기존에는 한글 슬롯만 확인).

## v2로 미룬 항목

### 1. 인라인 하이퍼링크 (`<href>`)

- **현재 동작:** HWP5/HWPX 하이퍼링크는 `Control::Field`(FieldType::Hyperlink)로
  도착하며 앵커 텍스트는 본문 런에 그대로 유지됩니다. URL/링크 의미만 `LossEntry`로
  기록되고 출력에는 평문으로 남습니다.
- **한계:** 크레이트 IR의 `Inline`에 `Href` 노드가 없습니다
  (`Text/Styled/FootnoteRef/LineBreak/Tab`만 존재). DocLang에는 `<href uri="">`가
  있으므로 표현 자체는 가능합니다.
- **v2 방향:** `Inline::Href { uri, content }`를 추가하고, HWP5 field 범위
  (`field_ranges`)를 이용해 본문 런의 정확한 문자 구간을 감싸 emit합니다.
  HWP3 경로는 URL 추출(rhwp upstream, 현재 `url = ""`)이 선행되어야 합니다.

### 2. 인텍스트 객체 위치 (in-text object positioning)

- **현재 동작:** 문단 텍스트 블록을 먼저 emit한 뒤, `controls[]`의 블록 객체를
  컨트롤 순서대로 뒤에 append합니다. 객체 간 순서는 보존되지만, 한 줄 텍스트 *중간*에
  박힌 객체의 정확한 인터리빙은 보존되지 않습니다.
- **한계:** rhwp가 제공하는 `field_ranges` / `control_text_positions()`를 아직
  사용하지 않습니다.
- **v2 방향:** 위 위치 정보로 텍스트 블록을 문자 오프셋 기준으로 분할하고 객체를
  실제 위치에 삽입합니다. (1)의 `<href>` 구간 처리와 동일한 기반을 공유합니다.

### 3. 미주(Endnote) 별도 표현

- **현재 동작:** DocLang 0.6에 `<endnote>`가 없어 `Block::Footnote`로 강등하고
  `LossEntry`를 남깁니다. 풋노트/미주 번호가 단일 순차 카운터로 병합됩니다.
- **v2 방향:** DocLang이 미주 요소를 도입하거나 preserve 경로에서 구분을 복원할 때
  별도 표현으로 분리합니다.

### 4. 좌표(`<location>`) 미부착 영역

- 글상자 내부 콘텐츠, 각주/미주 본문, 셀 내부 객체에는 좌표를 부착하지 않습니다.
  멀티페이지 블록은 첫 등장 페이지 박스만 기록합니다. 레이아웃 실패 시 빈 맵을
  반환합니다(현재 진단 로그 없음).
- 자세한 정책은 [README의 레이아웃 좌표 절](../README.md#레이아웃-좌표---location)을
  참고하세요.
